Platform Overview
Human evaluation with no DoorDash API dependency. Experts generate queries, search the live app as end users, manually capture top-5 results, and submit structured assessments optimized for AHT.

  
Roles
  • Expert ť create tasks, capture results from live app, evaluate, submit
  • Reviewer ť flagged evals + review queue
  • Admin ť all tasks, calibration feedback, delivery

  
Schema summary (AHT-optimized)
    
    
  • Capture: top 5 results ť restaurant name, cuisine, price, rating, delivery time (manual from live app)
    • 
    
  • Query-level: intent classification, complexity 1ť3, quality/diversity/ranking 1ť5, missing category flag, optional ambiguity + constraint tradeoff, optional overall feedback
    • 
    
  • Result-level (all 5): relevance 1ť5, placement (Should appear / Maybe / Should not appear)
    • 
    
  • Failure-only (relevance ? 2 or Should not appear): justification, failure tags, dropped constraints
    • 
  

  
Value beyond CTR / conversion
Behavioral metrics show what users click and order, not why results fail intent. This eval surfaces insights DoorDash likely under-captures today:
    
    
  • Constraint tradeoffs ť system satisfies price but drops dietary constraint (INTENT-PARTIAL + dropped-constraint tags)
    • 
    
  • Ambiguity handling ť how the system interprets vague or compound queries vs. what a user would reasonably expect
    • 
    
  • Ranking failures without engagement ť buried relevant results (RANK-BURIED) that never get clicks because they're below the fold
    • 
    
  • Diversity gaps ť redundant chains or missing subcategories in the result set (DIV-* tags)
    • 
    
  • False positives ť results that look plausible but violate query intent; these may get occasional clicks but erode trust over time
    • 
    
  • Context blindness ť ignoring time, location, or account tier signals (INTENT-IGNORE-CONTEXT)
    • 
  
The failure taxonomy turns these into structured, aggregatable signals ť not just anecdotal reviewer notes.






  
Expert workflow (no API)
  1. Create query in platform
  2. Open live DoorDash app (assigned market + account tier)
  3. Run query, capture top 5 results manually
  4. Rate relevance + placement for each
  5. Expand failure detail only on flagged results
  6. Submit ť auto-validation runs

  
Validation on submit
  1. All 5 results captured (restaurant name required)
  2. All 5: relevance + placement filled
  3. Failure results: justification + ?1 failure tag
  4. Query-level scales complete
  5. Consistency: high relevance + Should not appear, or low relevance + Should appear
